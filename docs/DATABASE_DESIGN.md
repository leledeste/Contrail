# Database Design

Status: implemented foundation, evolving design

This document defines the first SQLite database shape for the relay dashboard
and token-control plane. The initial migration and token importer are
implemented. The dashboard and production browser sessions are still future
work, and `.env` authentication remains the default until SQLite auth is
explicitly enabled.

## Goals

- Keep SQLite as the default self-host database.
- Let a relay owner create users and agent tokens from a dashboard.
- Store only hashes of secret tokens.
- Persist browser pairings and allow admin-side revocation.
- Record security-relevant actions in an audit log.
- Keep the local proxy offline/local; only the relay/dashboard uses the
  database.
- Preserve an `.env` fallback during migration so existing private relays do
  not break.

## Non-Goals

- Do not store IVAO credentials.
- Do not store voice audio.
- Do not store raw FSD or TS2 packets.
- Do not store chat history by default.
- Do not implement Shared Cockpit in this first schema.
- Do not build a full public SaaS account system in the first step.

## Storage Model

Default database path:

```text
/var/lib/contrail-relay/contrail.db
```

Local development can use:

```text
.contrail-relay/contrail.db
```

Recommended SQLite pragmas when opening the database:

```sql
PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = 5000;
```

WAL mode allows readers and one writer to coexist well enough for private and
small community relays. The relay should avoid high-frequency writes.

## SQLite Wrapper Decision

Preferred first implementation target: `better-sqlite3`.

Reasons:

- MIT license, compatible with Contrail's MIT license.
- Mature Node.js SQLite wrapper with a simple synchronous API.
- Good fit for the relay's expected low-frequency control-plane writes.
- Easier to reason about than an async wrapper for migration and dashboard
  admin operations.

Tradeoffs:

- It uses native bindings, so install/build behavior must be tested on Windows
  and Linux CI before release.
- Current versions require Node.js 20 or newer, which is now the project engine
  baseline.
- It should not be used for high-frequency audio or packet state. Live audio
  remains in memory/WebSocket only.

Alternatives reviewed:

- `node:sqlite`: attractive long-term because it removes a dependency, but it
  is not the first target while the Node.js API is still marked release
  candidate and requires a newer Node baseline.
- `sqlite3`: permissive license and async API, but more callback-oriented and
  less ergonomic for the relay's small admin/control database.

## Migration Table

```sql
CREATE TABLE schema_migrations (
  version INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  applied_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

Migrations should be plain SQL files applied in numeric order. The relay should
start only after all migrations succeed.

## Users

Users represent independent owners on the relay. In the current preview model,
one user owns one local agent and can connect from multiple browsers.

```sql
CREATE TABLE users (
  id TEXT PRIMARY KEY,
  username TEXT NOT NULL UNIQUE,
  display_name TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  disabled_at TEXT
);

CREATE INDEX idx_users_disabled_at ON users(disabled_at);
```

Notes:

- `id` should be generated by the server, not typed by the user.
- `username` is human-friendly and stable enough for the dashboard.
- Disabled users cannot authenticate new agent/browser connections.

## Agent Tokens

Agent tokens authenticate local Contrail proxies connecting outbound to the
relay. Tokens are shown once when created or rotated.

```sql
CREATE TABLE agent_tokens (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  token_prefix TEXT NOT NULL,
  token_hash TEXT NOT NULL UNIQUE,
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  last_used_at TEXT,
  revoked_at TEXT
);

CREATE INDEX idx_agent_tokens_user_id ON agent_tokens(user_id);
CREATE INDEX idx_agent_tokens_revoked_at ON agent_tokens(revoked_at);
CREATE INDEX idx_agent_tokens_last_used_at ON agent_tokens(last_used_at);
```

Notes:

- `token_hash` stores a one-way hash only.
- `token_prefix` is safe UI metadata such as the first 8 token characters. It
  helps identify a token without revealing it.
- `last_used_at` should be updated at most every few minutes per token to avoid
  unnecessary write load.
- Rotation creates or replaces token material and immediately revokes the old
  token after the relay reloads or after the dashboard commits the change.

## Agents

Agents are currently online WebSocket connections. Some state is live-only and
can stay in memory, but dashboard-friendly metadata belongs in the database.

```sql
CREATE TABLE agents (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  token_id TEXT REFERENCES agent_tokens(id) ON DELETE SET NULL,
  device_id TEXT NOT NULL,
  name TEXT NOT NULL,
  first_seen_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  last_seen_at TEXT,
  disabled_at TEXT,
  UNIQUE(user_id, device_id)
);

CREATE INDEX idx_agents_user_id ON agents(user_id);
CREATE INDEX idx_agents_last_seen_at ON agents(last_seen_at);
CREATE INDEX idx_agents_disabled_at ON agents(disabled_at);
```

Live status such as current WebSocket id, IP address, COM frequencies, callsign,
and TX readiness should remain in memory and be sent through the existing
remote protocol. The database only needs durable identity and history.

## Browser Pairings

Pairings authorize one browser id to control one agent. The browser id should be
hashed before storage.

```sql
CREATE TABLE browser_pairings (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  agent_id TEXT NOT NULL REFERENCES agents(id) ON DELETE CASCADE,
  browser_id_hash TEXT NOT NULL,
  browser_label TEXT,
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  last_used_at TEXT,
  revoked_at TEXT,
  UNIQUE(user_id, agent_id, browser_id_hash)
);

CREATE INDEX idx_browser_pairings_user_id ON browser_pairings(user_id);
CREATE INDEX idx_browser_pairings_agent_id ON browser_pairings(agent_id);
CREATE INDEX idx_browser_pairings_revoked_at ON browser_pairings(revoked_at);
```

Notes:

- Pairing codes remain short-lived and in memory.
- Durable pairings store only the hashed browser id.
- Admin-side revocation sets `revoked_at`; it should also disconnect or
  deauthorize matching live browser sockets.

## Browser Sessions

The first dashboard can start with a single admin password, but production
browser login needs server-side sessions.

```sql
CREATE TABLE browser_sessions (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  session_hash TEXT NOT NULL UNIQUE,
  browser_id_hash TEXT,
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  last_seen_at TEXT,
  expires_at TEXT NOT NULL,
  revoked_at TEXT,
  ip_address TEXT,
  user_agent TEXT
);

CREATE INDEX idx_browser_sessions_user_id ON browser_sessions(user_id);
CREATE INDEX idx_browser_sessions_expires_at ON browser_sessions(expires_at);
CREATE INDEX idx_browser_sessions_revoked_at ON browser_sessions(revoked_at);
```

This table can be deferred until the dashboard grows beyond a single admin
login, but reserving it in the design keeps the auth path clear.

## Audit Events

Audit events should be queryable for the common questions: who created or
rotated a token, who paired a browser, who revoked access, and which command
type was sent remotely.

```sql
CREATE TABLE audit_events (
  id TEXT PRIMARY KEY,
  user_id TEXT REFERENCES users(id) ON DELETE SET NULL,
  actor_type TEXT NOT NULL,
  actor_id TEXT,
  event_type TEXT NOT NULL,
  ip_address TEXT,
  target_agent_id TEXT,
  target_browser_pairing_id TEXT,
  command_type TEXT,
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  metadata_json TEXT
);

CREATE INDEX idx_audit_events_user_id ON audit_events(user_id);
CREATE INDEX idx_audit_events_event_type ON audit_events(event_type);
CREATE INDEX idx_audit_events_target_agent_id ON audit_events(target_agent_id);
CREATE INDEX idx_audit_events_command_type ON audit_events(command_type);
CREATE INDEX idx_audit_events_created_at ON audit_events(created_at);
```

`metadata_json` is acceptable only for extra details. Important fields must stay
as columns so the dashboard can search and filter audit history.

Suggested event types:

- `user.created`
- `user.disabled`
- `agent_token.created`
- `agent_token.rotated`
- `agent_token.revoked`
- `agent.connected`
- `agent.disconnected`
- `browser.paired`
- `browser.revoked`
- `remote_command.sent`
- `auth.failed`

## Token Hashing

Tokens should be generated from at least 32 random bytes and displayed as hex or
base64url. The server should store only a one-way hash.

Current implementation:

- Generate or recommend tokens from at least 32 random bytes.
- Store `SHA-256(token)` for preview relay tokens, not the raw token.
- Keep enough token prefix metadata for dashboard identification.
- Never log full tokens.

SHA-256 is acceptable here because relay tokens are long random secrets, not
human passwords. If Contrail later adds human passwords, those must use a
password hashing function such as Argon2id or bcrypt and a separate schema.

## Runtime Lookup

Implemented authentication path:

1. Browser or agent connects with a token.
2. Relay hashes the token and compares it with configured active token hashes.
3. If `CONTRAIL_RELAY_AUTH_MODE=sqlite-fallback`, active SQLite `agent_tokens`
   are loaded first, then `.env` tokens are added as fallback.
4. If `CONTRAIL_RELAY_AUTH_MODE=sqlite`, only active SQLite tokens are
   accepted.
5. Relay rejects disabled users or revoked tokens from SQLite.
6. Relay binds the WebSocket to the authenticated `user_id`.
7. Browser commands are routed only within the same `user_id`.

The existing `.env` token path remains during migration:

- `CONTRAIL_RELAY_AUTH_MODE=env` or unset: use only `CONTRAIL_RELAY_TOKEN`
  and `CONTRAIL_RELAY_USERS`.
- `CONTRAIL_RELAY_AUTH_MODE=sqlite-fallback`: use SQLite tokens plus `.env`
  fallback.
- `CONTRAIL_RELAY_AUTH_MODE=sqlite`: use only SQLite tokens.

Operational comparison:

| Mode | Token source | Operational benefit | Operational cost |
| --- | --- | --- | --- |
| `.env` only | Environment variables. | Best for the first boot because the owner can recover by editing one file. | Token rotation is manual and needs a relay restart; hard to build dashboard workflows on top of it. |
| SQLite with fallback | SQLite first, then environment variables. | Safe migration because a broken DB import does not immediately lock out existing users. | Two valid token sources increase audit and cleanup burden. |
| SQLite only | SQLite only. | Clean source of truth for future user dashboard, revocation, and audit features. | Requires database backup/restore discipline before it becomes the default. |

The intended migration order is deliberately conservative:

1. Keep `.env` auth as the default for private preview compatibility.
2. Import `.env` tokens into SQLite.
3. Run `CONTRAIL_RELAY_AUTH_MODE=sqlite-fallback` and confirm the imported tokens work.
4. Remove stale `.env` tokens after the database path is trusted.
5. Switch to `CONTRAIL_RELAY_AUTH_MODE=sqlite` for long-term operation.

## Env Import Helper

The first migration helper is:

```bash
npm run relay:db:import-users -- --env apps/relay/.env --db .contrail-relay/contrail.db --list
```

It reads:

- `CONTRAIL_RELAY_USERS`
- `CONTRAIL_RELAY_TOKEN`

and imports them into `users` and `agent_tokens`.

Import behavior:

- Same token for the same user: unchanged.
- Changed token for the same user: previous active token is revoked and a new
  token record is inserted.
- Legacy `CONTRAIL_RELAY_TOKEN`: imported as user `default` with token name
  `Legacy relay token`.
- Full tokens are never printed during import; only prefixes are shown.
- Relay authentication still uses `.env` by default.
- `CONTRAIL_RELAY_AUTH_MODE=sqlite-fallback` enables SQLite auth while keeping
  `.env` fallback for migration safety.
- `CONTRAIL_RELAY_AUTH_MODE=sqlite` disables `.env` fallback after DB
  tokens are confirmed to work.

For direct DB management without `.env`, use:

```bash
npm run relay:db:user -- list --db .contrail-relay/contrail.db
npm run relay:db:user -- add daniele --db .contrail-relay/contrail.db
npm run relay:db:user -- rotate daniele --db .contrail-relay/contrail.db
npm run relay:db:user -- revoke daniele --db .contrail-relay/contrail.db
npm run relay:db:user -- disable daniele --db .contrail-relay/contrail.db
npm run relay:db:user -- enable daniele --db .contrail-relay/contrail.db
npm run relay:db:user -- delete daniele --db .contrail-relay/contrail.db
npm run relay:db:user -- backup backup.db --db .contrail-relay/contrail.db
```

This CLI is the bridge between manual `.env` mode and the future dashboard. It
keeps the schema usable before a browser admin panel exists.

## Current Admin Panel

The first admin surface is intentionally small and token-protected:

- Served by the relay at `/admin`.
- Protected by `CONTRAIL_RELAY_ADMIN_TOKEN`.
- Available for `CONTRAIL_RELAY_AUTH_MODE=sqlite-fallback` and
  `CONTRAIL_RELAY_AUTH_MODE=sqlite`.
- Lists SQLite relay users.
- Creates users and shows the generated token once.
- Rotates/revokes tokens.
- Disables/enables users.
- Deletes users permanently.
- Lists known relay agents/devices and online status.
- Lists active browser pairings.
- Revokes browser pairings.
- Lists recent audit events.

It does not yet manage persistent browser sessions or multi-admin login. Those
remain dashboard work.

## Dashboard MVP

The first dashboard should stay small:

- Admin login.
- User list.
- Create/disable user.
- Create/rotate/revoke agent token.
- Online agent list. Implemented in the minimal admin panel.
- Browser pairing list. Implemented in the minimal admin panel.
- Revoke browser pairing. Implemented in the minimal admin panel.
- Audit event list. Implemented in the minimal admin panel for security/control
  events.

No Shared Cockpit controls should be added in this phase.

## Performance Notes

SQLite is appropriate for the expected write pattern:

- User and token changes are rare.
- Pairing and revocation are occasional.
- Audit events are low-frequency security/control events.
- Live audio and high-frequency state are not stored.

For a private or small community relay, SQLite should be sufficient. If Contrail
later runs a large public relay with many concurrent users and frequent audit
writes, PostgreSQL can be added as an optional backend without changing the
core data model.
